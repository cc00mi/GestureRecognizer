# 四号同学（9）— 项目工作总结报告

## 一、个人职责概述

根据项目分工，我主要负责系统显示界面相关工作，同时参与训练样本的制作与整理。我的工作重点是把识别模块输出的结果展示给用户，让系统不只是完成后台识别，而是能够在窗口中实时看到摄像头画面、手部检测信息和当前识别结果。

在项目中，我负责的显示部分位于 `ui/display.py`，并与 `main.py` 中的主循环进行对接。识别模块每次返回当前手势的 `label`、置信度 `score`、手部关键点 `landmarks` 等信息后，我的显示模块会把这些内容叠加到摄像头画面上，最终通过 OpenCV 窗口展示出来。

## 二、具体完成的工作

### 显示界面设计与实现 — `ui/display.py`

显示界面是整个手语识别系统面向用户的最后一环。摄像头模块负责采集画面，识别模块负责判断手势类别，而我的工作是把识别结果清楚地显示出来，让用户能够直接看到系统当前识别到了什么手语。

最开始的任务目标比较简单：窗口里能看到摄像头画面，并且能显示类似“当前识别：谢谢”的文字。根据这个目标，我采用 OpenCV 窗口作为显示方式，没有额外引入复杂的网页前端或 GUI 框架，这样可以减少部署难度，也更适合在树莓派环境中运行。

在 `ui/display.py` 中，我实现了 `render(frame, result, fps)` 函数。这个函数接收摄像头画面 `frame` 和识别模块返回的 `result`，然后在原始画面基础上绘制识别信息。`result` 中主要包含 `label`、`score`、`bbox`、`landmarks`、`message`、`handedness` 和 `type` 等字段。显示模块会优先取出 `label`，如果没有识别到有效手势，就显示“未识别”，避免界面出现空白或异常信息。

界面顶部设计了一个浅色信息面板，用来展示当前识别结果、置信度和 FPS。比如当识别模块返回 `label = "谢谢"` 时，画面顶部会显示：

```text
当前识别: 谢谢
置信度: 0.92   FPS: 28.6
```

这样用户在演示时可以一边看到实时摄像头画面，一边看到模型当前输出的识别结果。相比只在终端打印文字，这种方式更直观，也更适合课堂展示和项目答辩。

### 中文文字显示处理

OpenCV 自带的 `cv2.putText()` 对中文支持不好，如果直接绘制“当前识别：谢谢”，中文很容易显示成乱码或问号。因此我在显示模块中加入了 Pillow 中文渲染方案。

具体做法是先把 OpenCV 的 BGR 图像转换成 RGB 图像，再用 `PIL.ImageDraw` 绘制中文文字，最后转换回 OpenCV 使用的 BGR 格式。为了保证在 Windows 和 Linux / 树莓派上都能尽量正常显示中文，我设置了多个候选字体路径，包括：

```text
C:/Windows/Fonts/msyh.ttc
C:/Windows/Fonts/simhei.ttf
/usr/share/fonts/truetype/wqy/wqy-microhei.ttc
/usr/share/fonts/opentype/noto/NotoSansCJK-Regular.ttc
```

如果系统中安装了这些字体，界面就可以正常显示中文。如果 Pillow 或中文字体不可用，程序会退回到 OpenCV 的文字绘制方式，虽然中文显示效果会下降，但程序仍然可以继续运行，不会因为字体问题直接崩溃。

### 手部检测信息绘制

除了显示识别结果，我还在界面中加入了手部检测信息的绘制。识别模块或 MediaPipe 处理后会得到手部关键点，单只手包含 21 个关键点。我在 `ui/display.py` 中定义了 `HAND_CONNECTIONS`，按照 MediaPipe 手部骨架的连接关系，把拇指、食指、中指、无名指、小指和手掌部分连接起来。

绘制时先根据关键点坐标计算屏幕位置，如果坐标是 0 到 1 之间的归一化坐标，就按当前画面宽高转换成像素坐标。然后先画骨架连线，再画关键点圆点。这样做的好处是演示时可以直观看到系统是否正确检测到了手部位置，也方便调试模型识别不准确时的问题。

如果识别模块返回了 `bbox`，显示模块还会在画面中绘制手部检测框。这样用户可以同时看到检测区域、关键点骨架和最终识别文字，界面信息更完整。

### 与主程序和识别模块的对接

我的显示模块通过 `main.py` 与摄像头和识别模块完成对接。主程序的整体流程是：

```python
success, frame = camera.read()
result = recognizer_module.recognize(frame)
output = render(frame, result, fps=current_fps)
cv2.imshow(WINDOW_NAME, output)
```

这里 `camera.read()` 由摄像头模块提供，负责返回当前帧；`recognizer_module.recognize(frame)` 由识别模块提供，负责返回识别结果；我的 `render()` 函数负责把结果叠加到画面上。通过这种方式，各个模块之间的职责比较清楚，显示模块不需要关心模型内部如何推理，只需要按照约定读取 `result` 中的字段并完成展示。

为了配合演示，窗口使用 OpenCV 的 `cv2.imshow()` 展示，按 `q` 或 `ESC` 可以退出程序。主程序中还加入了 FPS 计算，显示模块会把 FPS 一起画到界面上，方便观察系统运行是否流畅。

### 异常和未识别状态处理

在实际运行中，摄像头可能会读取失败，或者画面中暂时没有检测到手势。为了避免界面显示混乱，我在显示模块中统一设置了默认状态。

当识别结果为空，或者没有有效 `label` 时，界面会显示“未识别”；当摄像头读取失败时，主程序会生成灰色空白画面，并传入类似下面的结果：

```python
result = {
    "label": "未识别",
    "score": 0.0,
    "message": "摄像头读取失败"
}
```

显示模块会把 `message` 作为状态提示绘制到界面上。这样即使硬件或摄像头暂时异常，用户也能从窗口中看到原因，而不是只看到程序卡住或黑屏。

### 训练样本制作参与

除了显示界面，我还参与了训练样本制作。项目需要识别 11 类中文手语词，包括：

```text
你好、再见、对不起、我爱你、没关系、谢谢、上课、下课、不舒服、厉害、多少钱
```

训练样本的质量会直接影响模型识别效果。我的工作主要是配合完成手语动作样本的采集、整理和检查，保证每个类别的动作样本尽量清晰、完整，标签和动作类别对应正确。

在制作样本时，需要注意手部是否完整出现在画面中，动作是否从开始到结束都比较清楚，背景是否会干扰手部关键点检测。因为模型后续会通过 MediaPipe 提取手部 21 个关键点，并把连续 30 帧组成一个动作序列，所以如果样本中手部被遮挡、动作过快或者标签错误，都会影响后续训练。

我参与样本制作的意义主要有两点。第一是补充训练数据，让模型能看到更多不同动作状态，提高泛化能力；第二是保证数据质量，减少错误标签和无效样本对模型训练的影响。项目 README 中显示，最终训练样本包括原始样本和自采集样本，增强后训练集达到 2604 条，验证准确率达到 96.51%。这说明样本制作和整理对最终识别效果起到了基础支撑作用。
